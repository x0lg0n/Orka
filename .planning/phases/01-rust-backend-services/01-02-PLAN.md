  <objective>Verify a Freighter session: the client signs a server-issued challenge with its Stellar key; backend verifies the ed25519 signature and enforces custody_mode='freighter'.</objective>
  <action>
    Implement `pub async fn issue_challenge(state: &AppState, stellar_address: &str) -> Result<String, AuthError>` returning a random server-generated challenge string (store ephemerally in a `DashMap<String,(String,Instant)>` keyed by address, TTL 5 min).
    Implement `pub async fn verify_freighter_session(state: &AppState, stellar_address: &str, signature_b64: &str, challenge: &str) -> Result<AuthenticatedUser, AuthError>`:
      - Decode the Stellar `G...` address via `stellar-strkey` (Ed25519PublicKey) to get the 32-byte ed25519 public key.
      - Verify `signature_b64` (base64) is a valid ed25519 signature over `challenge` bytes using `ed25519-dalek` `Verifier`.
      - Look up `profiles.custody_mode`; require == 'freighter' else `AuthError::WrongMode`.
      - Return `AuthenticatedUser` with the address.
    Wire `POST /auth/freighter/challenge` (issues) and `POST /auth/freighter/session` (verifies) in `routes()`.
  </action>

    - A wrong/challenge-mismatched signature returns `AuthError::InvalidToken`.
    - A 'orka' user presenting a Freighter proof returns `AuthError::WrongMode`.
  </acceptance_criteria>
